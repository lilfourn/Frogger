# Research: Tauri v2 macOS home directory access and AirDrop path semantics

## Summary
A Tauri v2 file manager can resolve the macOS home directory as `$HOME`, but reading it from the JS `@tauri-apps/plugin-fs` path requires both command permissions and a scope that matches the exact directory operation. A common self-inflicted failure is scoping only `$HOME/*` or `$HOME/**`: glob semantics mean `foo/**` matches sub-entries but not `foo` itself, so `readDir($HOME)` needs `$HOME` or the built-in home index/scope permissions as well. AirDrop is not a normal directory to list; Finder hosts a special AirDrop UI/app and outgoing file sharing should be handled via `NSSharingService`/share sheet, while received files usually land in Downloads.

Confidence: High for Tauri scope/glob behavior, Apple sandbox/TCC behavior, and AirDrop being non-directory; Medium for exact Tauri error-string mapping because app-specific wrappers may translate permission/scope failures into “path no longer exists.”

## Findings
1. **Tauri v2 fs access is capability + scope gated; `$HOME` support is explicit but not default.** The fs plugin requires paths to be in configured glob scopes and rejects out-of-scope API calls; available variables include `$HOME`. Default permissions only cover app-specific directories, not the whole home folder. Tauri’s autogenerated fs permissions include `fs:allow-home-read`, `fs:allow-home-read-recursive`, `fs:allow-home-meta`, `fs:allow-home-meta-recursive`, `fs:scope-home`, `fs:scope-home-recursive`, and `fs:scope-home-index`. [Tauri fs JS docs](https://tauri.app/reference/javascript/fs), [fs permissions reference](https://github.com/tauri-apps/plugins-workspace/blob/v2/plugins/fs/permissions/autogenerated/reference.md)

2. **`$HOME`, `$HOME/*`, and `$HOME/**` do not mean the same thing.** Tauri scopes are glob patterns. The underlying Rust glob semantics document that a pattern ending in `/**` matches sub-entries such as `foo/a` and `foo/a/b`, **but not `foo` itself**; similarly `$HOME/*` matches direct children, not the home directory path. Therefore `readDir(home)` or `readDir('', { baseDir: Home })` needs the directory itself allowed (`$HOME` or `fs:scope-home-index`/`fs:scope-home`) plus the `read_dir`/metadata command permission; recursive browsing also needs children (`$HOME/*`, `$HOME/**` or `fs:scope-home-recursive`). [globset syntax](https://docs.rs/globset/latest/globset/), [Tauri fs permissions reference](https://github.com/tauri-apps/plugins-workspace/blob/v2/plugins/fs/permissions/autogenerated/reference.md)

3. **Recent Tauri discussion confirms confusion around `**` vs `**/*`.** A 2025 Tauri plugins PR noted that docs using `**` were confusing and corrected an example toward `**/*`; this supports treating scope patterns carefully and testing the exact target path. [Tauri plugins PR #2969](https://github.com/tauri-apps/plugins-workspace/pull/2969)

4. **`tauri::path::PathResolver::home_dir()` only resolves a path; it does not grant read access.** Tauri’s Rust path resolver says `home_dir()` resolves to `$HOME` on macOS. Rust `std::fs::read_dir` then performs the real OS operation and can fail if the path does not exist, permissions are insufficient, or the path is not a directory. So compare: (a) can the app resolve home? (b) does Tauri fs scope permit it? (c) does macOS permit the process to list it? [Tauri PathResolver docs](https://docs.rs/tauri/2.0.0/tauri/path/struct.PathResolver.html), [Rust `std::fs::read_dir`](https://doc.rust-lang.org/std/fs/fn.read_dir.html)

5. **macOS TCC/FDA protects specific folders and full-disk locations; expected OS errors are permission-like, not true absence.** Apple states macOS 10.15+ requires user consent for Documents, Downloads, Desktop, iCloud Drive, and network volumes; Full Disk Access must be granted by the user in System Settings for broad storage access. Apple’s sandbox diagnostics list POSIX/ACL denial as `EACCES` and SIP/data-protection denial as `EPERM`. Thus a raw OS denial usually manifests as “permission denied”/“operation not permitted,” although higher-level app code may map a denied/failed listing to “unavailable” or “path no longer exists.” [Apple Platform Security: controlling app access](https://support.apple.com/en-ca/guide/security/secddd1d86a6/web), [Apple sandbox file access docs](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)

6. **App Sandbox changes the model: arbitrary `~` traversal is not automatically allowed.** A sandboxed macOS app gets unrestricted access to its container under `~/Library/Containers`, and access outside the sandbox comes from user-selected files/folders, security-scoped bookmarks, selected standard-folder entitlements, or user-granted Full Disk Access. A folder selected through a standard open panel extends sandbox access recursively to its contents, but some items can still fail for POSIX/TCC/SIP reasons. [Apple sandbox file access docs](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)

7. **Full Disk Access has no entitlement/code shortcut.** Apple explicitly says an app cannot automatically gain Full Disk Access through an entitlement or code; the user must grant it in System Settings > Privacy & Security. For a file manager, defensive error handling and a guided “grant FDA” flow are expected if you need broad traversal of protected home subtrees. [Apple sandbox file access docs](https://developer.apple.com/documentation/security/accessing-files-from-the-macos-app-sandbox)

8. **AirDrop is a Finder sharing UI, not a filesystem directory.** Apple’s user docs describe opening AirDrop in Finder and dragging items to a device; received files on Mac are generally saved to Downloads or sometimes opened/saved by the receiving app. Apple’s developer API for sending by AirDrop is `NSSharingService`, which “displays a sheet” and can “send a file using AirDrop”—not a directory listing API. [Apple AirDrop user guide](https://support.apple.com/guide/mac-help/send-files-using-airdrop-mh35868/mac), [NSSharingService docs](https://developer.apple.com/documentation/appkit/nssharingservice)

9. **The Finder AirDrop sidebar item has an app bundle target, not a user-folder path.** Community evidence for programmatically opening Finder’s AirDrop points to `/System/Library/CoreServices/Finder.app/Contents/Applications/AirDrop.app`; this is a hidden Finder component, not a location under `~` that a third-party file manager can `read_dir`. Treat this as implementation detail: useful for “open Finder AirDrop,” not for listing. [Stack Overflow: Open AirDrop programmatically](https://stackoverflow.com/questions/55249863/open-airdrop-programmatically)

10. **Other cross-platform/Tauri file managers generally omit AirDrop or treat it as a special action.** Search results for Tauri/Electron file managers show typical sidebars listing real folders (Desktop, Documents, Downloads, Pictures, volumes) rather than AirDrop; no strong evidence of a Tauri/Electron app listing AirDrop as a filesystem directory. Direct implication: if the app sidebar includes AirDrop, clicking it should not concatenate it with Home or call `read_dir($HOME/AirDrop)`; it should open Finder’s AirDrop UI or trigger a native share/AirDrop action for selected files.

## Direct debugging implications
- For “home directory unavailable/path no longer exists”: first log the exact path passed to `readDir`/Rust `read_dir`, the Tauri command permission, and the scope match. If using plugin-fs from JS, include the home directory itself (`$HOME`) in addition to child patterns (`$HOME/*`, `$HOME/**`) or use `fs:scope-home-index`/`fs:scope-home(-recursive)` plus `fs:allow-read-dir`/metadata permissions.
- If `std::fs::read_dir(app.path().home_dir()?)` in a Rust command fails, Tauri fs scope is not involved; inspect OS error kind/raw_os_error. `EACCES`/`EPERM` points to macOS permissions/TCC/sandbox/FDA, while `ENOENT` suggests wrong path construction, stale symlink, race, or app code mapping.
- On macOS 14/15, expect prompts/Settings controls primarily for protected subfolders (Desktop/Documents/Downloads, iCloud, network/removable volumes) or app-container access; the top-level `/Users/<name>` listing should usually work for a non-sandboxed/notarized app unless Tauri scope or POSIX permissions block it.
- For “AirDrop selected with Home”: this is almost certainly a routing bug. AirDrop should be a virtual sidebar action, not a child entry of Home and not a readable path.

## Gaps
- I did not find a primary Apple statement saying “AirDrop is not a filesystem directory”; this conclusion is inferred from Apple documenting AirDrop as Finder/share-service UI plus the absence of any filesystem listing API/path.
- Exact wording “path no longer exists” could come from the app’s own error normalization, Tauri/plugin error conversion, or a UI library. Next step: capture raw Rust `io::ErrorKind`/`raw_os_error` and raw JS plugin rejection before mapping to UI text.
